UI/UX for FC estimate builder

Either upload admission note or type what the doctor has said about the patient
Ask patient payor type
If insurance, fill out only the required insurance details 
Put a flag for only treatments where there maybe a strong likelihood of sublimits. AI should be able to easily classify the entire list for you one time and populate it.
Check if there's any package available based on the selected payer and treatment. If there is No package, then tell the user that it would be an open billing. If there is a package, then Pull all the necessary details from the package. In case the system does not have all the necessary details of the package, tell the user that there is a package available, but the system does not have this necessary information. In this case, Allow user to go ahead with open billing format.
Show the final estimate built by the system, Allowing the user to delete any rows, change any amount as they see fit.
If it's a package billing, the user should have A single click option to remove all the extras over the package that are being charged. 
Users should be able to also see the historic PF P50 values for that set of cohorts. While in package billing, the PF is absorbed in the package, In non-package billing If the PF calculated via logic and the historic PF are significantly different, then the historic PF P50 should be used to give an estimate.
The main estimate workbook view should have all the primary buckets and their values, and then a drop-down where, if they click on the drop-down, they can see all the line items in that. That will keep the UI simple and user-friendly when they want to drill down.
The main estimate workbook should be very neat and clean and tidy, with no complexity. There should be two buttons: Advanced Controls, Historic Metric, Both have been described below.
If they don't want to use a system-generated one, Or our system is not capable of generating one because it doesn't have the past template or it doesn't have the package details, Or it's not functioning properly etc., they should have a very, very, very simple form like Estimate Builder, which has only the key buckets and the user can just put the amount and that's it. This will ensure a safe fall back for the user To reliably use in case the system ever doesn't work. 
The downgrade and upgrade option in the next step should only be given if the system knows the Estimate amount for the downgrade/upgrade option. If not, then it should be greyed out.

Advanced control:

Select Estimate Mode - Low, typical, or high
The way low typical high works is that low/typical/high uses p25/p50/p75 values for quantities Without changing the entered LOS and OT hours. So, based on the selection, it uses the right pharmacy numbers and service quantity numbers.
OT consumables, advanced control where they can select a couple of items and change the number accordingly - already present in Excel
Service add-ons & Group add-ons in the same view
Implants (P25,P50,P75 Values) at the bucket level. No need for implant-wise drill down like it’s present in Excel. 


_________

Based on the presence rate, if it's 90% up then robotic charges should automatically be applied. If it's less than 90% then the user should be prompted and asked if it's a robotic surgery or not On the same estimate workbook page. 

_________

Historic Metrics:
Historic metric shows the relevant p50 values, p25 values, p75 values of the relevant buckets and the gross total And similarly, for the pharmacy, sub-bucket as well. It only shows very relevant information and not too sparse information. For example, it should not drilling down into each items, etc. It should just show that:
Pharmacy is typically from this range to this range.
Professional fees from this range to this range.
OT hours
OT consumables are generally from this range to this range.
OP/IP is generally from this range to this range.
Investigations are from this range to this range.
And it should clearly say if the historic metrics that are being shown are for cash, Gipsa, non-Gipsa, all insurance (Gipsa + non-Gipsa), corporate or all payers. And the historic case count against it
While clearly comparing these numbers with the selected or shown values on the estimate
