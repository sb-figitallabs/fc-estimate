# How NME (Non-Medical Expenses) Is Derived

## Plain-language answer (for the PM)

We do not look up NME from any hospital billing data or a master list of non-payable items. In the full estimate flow, NME is simply the sum of seven specific service lines that we always mark as never-insurer-payable — drug administration, medical records, ward consumables, dressing, diet consultation, MLC charges, and warmer — whatever those lines happen to cost in that estimate; nothing else in the bill (including gloves, kits, or any pharmacy consumable) is ever counted as NME. In the simple/manual flow there is no derivation at all: the financial counselor types the NME amount into a box and we pass it through as 100% patient-payable.

---

## How it works

There are two live paths and one dead-end (a database column that exists but is never read).

### Path A — Full estimate (itemized settlement)

1. **Line items are generated by the engine**, not read from a bill. Driver rows are priced as tariff rate × cohort-derived days from `fc.service_tariff_rate_matrix` (fetched in `src/modules/engine/artifacts.js:311`), and history rows come from cohort quartiles over `mart.main_table` (`src/modules/engine/artifacts.js:32-53`). The rows that will later count as NME are created in `src/modules/engine/lineItems.js`:
   - `Ward Consumables` — tariff code `HSP5013` × LOS days (lines 296, 391)
   - `MLC Charges` — tariff code `HSP0047`, only when the MLC input is Yes (lines 298-313, 409)
   - `Medical Records` / `Medical Records - 1 Day` — codes `RNS0120` / `MSC10` (lines 232-234, 348)
   - `Warmer` (`EME0087`), `Diet Consultation` (`DIE0001`), `Dressing - Minor` (`CAS0007`) — fixed template rows (lines 392-396)
   - `Drug Administration Charges` — **computed as 12.5% of the pharmacy total**, no tariff code (line 424)

2. **Classification happens at settlement time by a hardcoded name regex.** `src/modules/insurance/settlement.js:23`:

   ```js
   const NME_ROWS = /DRUG ADMINISTRATION|MEDICAL RECORDS|WARD CONSUMABLES|DRESSING|DIET CONSULTATION|MLC CHARGES|WARMER/i;
   ```

   `classifyRow()` (`settlement.js:26-33`) tags any line item whose *name* matches this regex as class `nme`. The comment calls it the "KIMS NME list".

3. **NME rows get admissible = 0** (`settlement.js:168`) — the insurer pays nothing on them — and are excluded from sub-limit pools (`settlement.js:185-186`). The patient-side NME figure is just the raw sum of those rows (`settlement.js:203`), surfaced as `patient.nme` (`settlement.js:215`) alongside copay, proportionate deduction, sub-limit overflow, room-upgrade excess and beyond-cover (`settlement.js:214-222`).

4. **Where it surfaces in the API response:** `settle()` is called in `src/modules/engine/buildEstimate.js:486` → `estimate.insurance_settlement.patient.nme`; per-room replay at `buildEstimate.js:561-566` → `estimate.by_room.<room>.settlement.patient.nme`. The package route (`settleWithPackage`, `settlement.js:256-308`) settles only the payable extras, so its NME reflects extras only — the frontend deliberately avoids it (see below).

### Path B — Simple/manual flow (bucket-level)

`POST /api/estimate/settle-manual` (`src/routes/estimate.routes.js:191-204`) accepts an explicit `nme_amount` (zod, default 0). `settleManual()` (`settlement.js:336-353`) appends it as a synthetic row `Non-medical (NME)` with `admissible: 0` and echoes it back as `patient.nme` (`settlement.js:404`). **The number is whatever the FC typed** — the Hospital_OS Simple form has an NME input defaulting to '0' (`frontend/src/modules/estimate-builder-v3/components/SimplePage.tsx:56, 107, 262`).

### Path C — The database column nobody reads

`fc.package_bill_admissions.nme_amount` exists (created and populated from the HIMS pkg_detl export by `scripts/load-package-bills.js:391, 442`) but **no serving code ever queries it**. The billed-actuals quartiles use `final_pkg_bill_excl_fnb` and `pkg_gross_amount` instead (`src/modules/packages/packages.service.js:256-294`, `src/routes/lookup.routes.js:282-299`). `mart.main_table` has no NME column either — the cohort fetch (`artifacts.js:32-53`) selects bucket totals, stays and pharmacy JSON, nothing NME-labelled.

### Frontend consumption (Hospital_OS, estimate-builder-v3)

- `components/WorkbenchScreen.tsx:507` — per-room patient-share chip `NME ₹x` from `settlement.patient.nme`.
- `components/InsurancePanel.tsx:168` — "Non-payable items ₹x" line from `patient.nme`.
- `components/PreviewScreen.tsx:128-161, 294-296` — patient-facing "Not covered (NME)" figure and the out-of-pocket reason line ("non-medical items"). Note the comment at lines 130-144: the preview intentionally uses the itemized `insurance_settlement` (not the package settlement) because the package settlement under-reports NME (would show ₹0 where the itemized path shows e.g. ₹2,970).
- `components/SimplePage.tsx:56-262` — manual NME input posted as `nme_amount`.
- `lib/fcApi.ts:140, 148, 177, 187` — response typings carrying `patient.nme` / `nme_amount`.

## Data sources

| Table / column / input | Role in NME |
|---|---|
| `fc.service_tariff_rate_matrix` (rate per item_code per room) | Prices the seven NME-classed service rows (e.g. Ward Consumables `HSP5013` × LOS days, MLC `HSP0047`, Medical Records `RNS0120`/`MSC10`, Warmer `EME0087`, Diet Consultation `DIE0001`, Dressing `CAS0007`) |
| `mart.main_table` (cohort cases) | Supplies LOS/ICU-day drivers and pharmacy quartiles that scale those rows (and the pharmacy total that Drug Administration = 12.5% of); has **no** NME column of its own |
| Hardcoded regex `NME_ROWS` in `settlement.js:23` | The actual "NME rule" — 7 item-name patterns, 100% patient, insurer-admissible ₹0 |
| Request body `nme_amount` (`/api/estimate/settle-manual`) | Manual flow: FC-entered amount, passed through untouched |
| `fc.package_bill_admissions.nme_amount` | Loaded from the HIMS export but **never queried** by any API — dead data today |
| API output `insurance_settlement.patient.nme`, `by_room.<room>.settlement.patient.nme`, `settle-manual → patient.nme` | Where the figure surfaces to the frontend |

## Caveats (the blunt version)

1. **NME is a hardcoded 7-name list, not data-driven.** Anything not literally named DRUG ADMINISTRATION / MEDICAL RECORDS / WARD CONSUMABLES / DRESSING / DIET CONSULTATION / MLC CHARGES / WARMER is not NME. The IRDAI non-payables list has ~190 items; we model 7 service lines. Renaming a service breaks the match silently.
2. **Classic NME items (gloves, admission kit, disposables) inside Pharmacy are treated as fully insurer-payable.** `classifyRow()` classes the entire Pharmacy bucket (including IP/OT consumables) as `exempt` — full admissible, no deduction (`settlement.js:29`). So the engine's "NME" is smaller than a hospital-finance definition of NME; the real consumables spend is assumed covered.
3. **Drug Administration Charges is itself a hardcoded 12.5%-of-pharmacy rule** (`lineItems.js:424`) — so part of the NME figure is a percentage of an estimate, twice removed from any actual non-payable-item data.
4. **Manual flow NME is pure user input.** No validation beyond non-negative; if the FC leaves it at 0, the estimate implies zero non-payables.
5. **The one real historical NME signal we have — `fc.package_bill_admissions.nme_amount` from actual converted bills — is loaded and ignored.** It could ground-truth or replace the regex approach but nothing reads it.
6. **Package-route settlement under-reports NME** (extras-only); the frontend works around this by always displaying the itemized settlement's figure (`PreviewScreen.tsx:128-144`). The workaround lives in the UI, not the engine.
